## Executive Summary

**Verdict: YES — this is a strong open-source project.** The AI code generation + PR automation pipeline is a genuinely novel gap in the OSS landscape. No existing tool (HolmesGPT, k8sgpt, StackStorm, Keep) does the full loop. The event-driven plugin architecture is the right approach. The CODING-STANDARDS.md is enterprise-grade.

## Architecture Decision: Event-Driven Plugin Architecture

Chosen over Monolith-First (too coupled) and Microservices (GCP-locked, too heavy).

```
sentinelai/
├── core/           # ~700 LOC - event bus + SQLite outbox, plugin registry, orchestrator
├── contracts/      # Abstract interfaces per pipeline stage
├── plugins/        # Reference implementations (monorepo for v1)
├── cli/            # Rich terminal CLI
└── api/            # Deferred
```

### Event Flow
```
AlertDetected ──► TriageComplete ──► TicketCreated
     │                                    │
     └──────── SQLite outbox ─────────────┘
               (durable, replayable)
                                          [Phase 2+]
     FixGenerated ──► PROpened ──► [Human Gate] ──► PRMerged ──► DeployStarted ──► TicketClosed
```

### Key Constraints (from CODING-STANDARDS.md)
- Clean architecture: domain → application → infrastructure → transport (rule 3)
- Every failure named and handled explicitly (rule 4)
- Structured observability on all events (rule 5)
- Bounded external calls with configurable timeouts + circuit breakers (rule 6)
- Configuration external, validated at startup (rule 7)
- Formal OpenAPI contract for webhook receiver (rule 10)
- SQLite outbox for event durability (rule 9 compliance)

## Revised Phase Plan (post-outside-voice)

### Phase 1: Lean Core (2 weeks)
**Goal:** Prove AI triage works. Minimal surface area.
- Core event bus + SQLite outbox + plugin registry
- `file_source` plugin (read alerts from JSON files)
- Claude triage plugin
- Console output (rich terminal via `rich` library)
- CLI: `sentinelai triage --file`, `sentinelai demo`, `sentinelai doctor`
- Error hierarchy + config validation
- **No Jira, no Gemini, no webhook** — deferred to Phase 1.5

### Phase 1.5: Integrations (2 weeks)
- Webhook alert source (HMAC-SHA256 auth)
- Jira ticket plugin + GitHub Issues plugin
- Gemini triage plugin
- Incident timeline (JSONL audit trail)
- `sentinelai explain`, `sentinelai costs`, `sentinelai timeline`
- `sentinelai plugin new` scaffold generator
- OpenAPI spec for webhook receiver

### Phase 2: The Differentiator (4 weeks)
- CodeFixer contract + Claude code fixer plugin
- Fault-domain file selection (extend TriageComplete with `affected_files`)
- PR automation via GitHub API
- Human approval webhook gate
- Jira comment integration with AI reasoning chain

### Phase 3: Full Loop (4 weeks)
- Deployer contract + Cloud Run reference plugin
- Validator contract + Playwright/pytest reference plugins
- Blue/green deployment orchestration
- Auto-close ticket on validation pass

**Total: ~12 weeks human / ~2 weeks CC+gstack**
